# Q1 non-LLM recipe-search protocol audit

Status: **design audit; no formal Q1 result is authorized**

Date: 2026-07-20

API increment: **0**

## Question

Can the frozen PU evidence already support a causal claim that the LLM proposal
policy is better than a strong non-LLM policy under an equal search budget?

## Audited evidence

The counts below are generated by
`breeze/scripts/audit_q1_nonllm_budget.py`; the structured output and aggregate
input-manifest hashes are in
`analysis/q1_nonllm_budget_audit_2026-07-20.json`.

- The archived LLM run contains exactly 450 class-balanced slots and 922 saved
  recipe candidates. The class totals are 301 candidates for healthy, 238 for
  OR, and 383 for IR. The v2 rescreen admits 286 slots.
- Its cumulative admitted-slot counts are 205 at `K=0`, 241 at `K=1`, 268 at
  `K=2`, and 286 at `K=3`, as frozen in
  `breeze/results/admission_round_freeze_2026-07-17/`.
- The legacy random-v2 run contains 450 one-shot slots and admits none under the
  v2 verifier.
- The legacy rule-v2 directory contains 700 one-shot slots, because IR was
  extended from 150 to 400 slots to obtain enough accepted samples. It admits
  204 slots in total. Its final balanced downstream pool therefore matches the
  accepted-sample budget, but not the proposal budget of the archived LLM run.
- `breeze/scripts/recipe_ablation.py` already contains a deterministic,
  train-only `empirical_recipe` candidate. It samples class-conditional training
  statistics and maps them to the existing renderer schema. However, the CLI
  does not expose this source, no empirical pool exists, and no formal validity,
  budget, independence, or downstream audit has been completed.
- No Bayesian/TPE optimizer or general verifier-feedback controller is present
  in the repository. Optuna, scikit-optimize, and Hyperopt are not installed in
  the required `.venv-breeze` environment.

## Decision

The current fixed-pool results support the manuscript's narrow source-ablation
statement (LLM, rule, and open-loop random under the frozen legacy protocol).
They do **not** support the stronger Q1 claim that LLM feedback beats the
strongest non-LLM search under an identical proposal budget. In particular,
`450 slots` is not an equal query budget: the archived LLM policy consumed 922
saved recipe proposals, whereas legacy rule and random consumed one proposal per
slot, and the rule pool required 700 slots.

## Protocol required before execution

1. Define one legal recipe domain shared by random, rule, train-stat estimator,
   deterministic rule-feedback, and TPE. Every bound must follow from renderer
   validity, kinematics, or outer-train calibration; no outer-test or downstream
   metric may set a bound.
2. Define a continuous train-only nonconformity objective directly from the
   frozen verifier report. The primary comparison remains binary admission;
   optimizer ranking must not use downstream accuracy or outer-test data.
3. Count both recipe proposals and renderer/verifier evaluations. Expansion
   seeds are reported separately and cannot be silently treated as free search
   evaluations.
4. Give every policy the same per-class proposal-evaluation budget and the same
   renderer-seed schedule. Preserve every rejected candidate, recipe, seed,
   report, runtime, and hash.
5. Run a one-class, five-candidate smoke first. Smoke values cannot enter a
   claim. Freeze the pilot capacity rule before any full pool.
6. Treat independently generated pools as the first-level unit. The existing
   one archived LLM pool cannot become independent-pool evidence by adding CNN
   seeds. New LLM pools require a separate, explicit API authorization.

## Immediate zero-API work allowed

- expose and unit-test the train-stat estimator in a new non-formal output root;
- implement a general recipe-domain validator and a verifier-only objective;
- implement and test rule-feedback and TPE with complete evaluation ledgers;
- run bounded train-only smoke/pilot work without reading outer-test metrics;
- calculate the exact API, compute, and storage request for matched independent
  pools, but do not issue provider calls.

## Train-stat estimator smoke

The previously unreachable empirical source is now exposed behind the explicit
tagged CLI path without changing the legacy `both` default. A bounded healthy-
class smoke used five one-shot candidates in
`breeze/runs/recipe_ablation_empirical_v2_q1_empirical_smoke_2026-07-20/`.
It admitted 2/5 slots and produced nine verifier-feasible items after expansion
and diversity admission. These values are interface diagnostics only and are
not evidence for a paper claim.

`breeze/scripts/audit_recipe_ablation_run.py` independently regenerated every
primary and expansion waveform from the archived recipe and seed, replayed all
gate decisions, and rebuilt the pool exactly. The structured audit, including
the verifier hash and run-manifest hash, is
`analysis/q1_empirical_smoke_audit_2026-07-20.json`. The audit loader requests
only `load_file_split('train')`.

## Verifier-only optimizer objective

`breeze/src/recipe_search.py` now implements the part of the optimizer protocol
that can be fixed without selecting a recipe search range. It maps every active
v2 hard constraint to a dimensionless threshold- or interval-normalized
violation and returns their L-infinity norm. Thus zero is equivalent to binary
admission, no hand-tuned cross-gate weights are introduced, and a failing
unknown gate raises instead of being ignored. Fault-envelope alternatives use
the least-violating calibrated resonance band because the verifier itself is an
`any-band` gate. The function accepts only a verifier report, its train-only
calibration artifact, and the class; it has no downstream or test interface.

Five focused tests cover deterministic/renderable empirical proposals,
normalization, disjunctive constraints, feasible-zero equivalence, and
fail-closed handling. On the five-candidate empirical smoke, the two admitted
candidates score exactly zero and the three rejected candidates score strictly
above zero. This freezes the feedback objective but **not** the TPE recipe
domain or proposal budget.
